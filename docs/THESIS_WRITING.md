# 毕设论文写作素材（可直接改写入论文）

本文件聚焦“写论文要用的技术描述”，避免与 `docs/ARCHITECTURE.md` 重复的实现细节。

---

## 1. 研究背景与问题定义（可作绪论/需求分析）

在企业/组织知识管理场景中，文档分散、更新频繁、结构不统一，传统关键词检索难以满足“自然语言问答 + 可追溯依据”的交互需求。大模型具备强大的语言生成能力，但在缺少外部约束时可能产生“幻觉”，即生成缺乏事实依据的内容。为兼顾回答质量与可控性，本项目采用检索增强生成（RAG, Retrieval-Augmented Generation）架构：先从知识库中检索与问题相关的文本片段，再将片段作为上下文约束模型生成，从而降低幻觉并提升可解释性。

---

## 2. 系统总体架构（可作系统设计）

系统由前端静态页面与后端 FastAPI 服务组成：

- **前端**：负责对话交互、文档上传、知识组选择、配置管理（API Key/模型等保存在 LocalStorage，每次请求随 `settings` 发送）。
- **后端**：负责文档解析（PDF 优先 OpenDataLoader v2+，回退 PyMuPDF；DOCX 用 `python-docx`）、Milvus 向量索引构建、检索加权与重排、模型推理（流式输出）、PostgreSQL 引用反查、评估接口。

为满足“全在线”与可答辩性：

- Embedding、Reranker、LLM、视觉解析均可使用 **硅基流动** 的 OpenAI 兼容接口；
- 同时提供 **Ollama 兜底**（同样走 OpenAI 兼容协议，仅更换 base_url），便于答辩说明“可本地部署/数据不外传”的可行性。

---

## 3. 知识组织：Milvus 向量库 + PostgreSQL 业务表 + 知识组映射（可作数据结构设计）

本项目采用“**Milvus 向量库 + PostgreSQL 业务表（SQLModel）**”：

- **向量索引（Milvus）**：文档切 chunk 后写入同一 collection；字段含 `doc_id`（及切分序号等），便于检索后反查业务表。
- **PostgreSQL**：存储文档全文与文件名、知识组与多对多映射、对话记忆；引用阶段用 `doc_id` 主键反查文档名，避免仅依赖模型“记住”文件名。
- **知识组（Groups）**：仅存于 PostgreSQL，不向向量库写入 `group_id`；查询时先召回再按组过滤与加权。

这样可以在不增加向量库复杂度的前提下实现：

- 文档可属于多个组；
- 一次查询可选择多个组；
- 通过组优先级（主/次组）影响检索排序。

---

## 4. 检索与排序：多组加权 + 在线 Reranker（可作算法流程）

一次查询的检索流程如下：

1. **向量召回**：对问题做向量化，在 Milvus 全局索引中召回 topK 候选 chunk；
2. **按知识组过滤**：若用户选择了知识组，则仅保留 `doc_id` 属于所选组的候选；
3. **组权重加权**：对每个候选 chunk 的分数进行加权，体现主/次组优先级；
4. **在线 Reranker 精排**：将候选 chunk 交给在线 rerank 模型重排，取 topN 作为最终上下文；
5. **生成**：将上下文与问题送入 LLM；文末由系统根据本轮 `doc_id` 集合自 PostgreSQL 追加「引用文档」列表，与纯提示词约束形成互补。

**引用与可解释性（防幻觉）**：检索片段在 prompt 中带 `[来源i]` 与 `doc_id`；回答结束后用 `doc_id` 回查 PostgreSQL 得文档名；流式/非流式均在文本末输出引用块，且 JSON 可返回 `sources`。

---

## 5. 多智能体：Agno Team + 路由与混合输出（可作创新点/扩展设计）

为提高跨领域问答的稳定性，本项目引入 Agno 框架的 **Team** 能力，将回答拆分为三个角色：

- **医疗 Agent**：用“提示词模拟微调”约束医疗安全与风险提示，减少不当建议；
- **法律 Agent**：用“提示词模拟微调”约束法学分析结构（要件/程序/前提/风险）；
- **协调者（Team leader）**：根据问题进行路由，决定调用单一成员还是双成员并融合输出。

路由策略采用“启发式关键词 + 协调者融合”的轻量方案：命中单领域关键词走单输出，命中双领域走混合输出；并在最终答案顶部输出 `【路由说明】...`，便于实验记录与答辩展示。

---

## 6. 工程可靠性：单例模式（可作工程实现/性能优化）

为减少重复初始化开销并保证全局一致性，本项目对“模型客户端/Team 构建器”引入单例模式：

- 使用 `@singleton` 装饰器实现“按配置缓存”的全局唯一实例；
- 通过线程锁保证并发下的安全初始化；
- 使得在高频请求下复用模型对象/Team 结构，降低延迟与资源消耗。

---

## 7. 评估设计（可作实验设计）

项目提供轻量评估接口 `POST /evaluate`，对每条样例输出：

- 检索上下文 `context`（便于人工核查）
- 生成答案 `answer`
- 语义相似度 `semantic_similarity`：使用同一在线 embedding 模型计算 `answer` 与 `expected` 的余弦相似度，作为“语义相关性 proxy”

该方案实现简单、可复现、便于展示“模型/参数/策略”的对比实验（例如不同 embedding/reranker、不同 topK/topN、单组 vs 多组加权、不同路由策略等）。
